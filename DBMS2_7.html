DBMS II: Page 7 - Performance Tuning: Query Optimization & Execution Plans

The Need for Speed

In a small database with 100 rows, every query is fast. But in a production environment with 100 million rows, a poorly written query can take minutes or even hours to complete, effectively "locking" the database and crashing your application. Performance Tuning is the art and science of optimizing database structure and SQL queries to ensure the fastest possible response times. This page explores the Cost-Based Optimizer, how to read an Execution Plan, and the advanced indexing strategies used by professional DBAs to keep systems running smoothly under heavy load.

1. The Cost-Based Optimizer (CBO)

When you send a query to the DBMS, it doesn't just run it. A component called the Optimizer looks at many different ways to execute the query (e.g., using an index vs. a full table scan). It calculates a "Cost" for each path based on statistics (how many rows are in the table, how unique the values are). The DBMS then chooses the path with the Lowest Cost. As an engineer, your job is to provide the optimizer with the right tools (Indexes) and the right information (Updated Statistics) to make the best choice.

2. Reading the Execution Plan: EXPLAIN

The most important tool in your tuning kit is the EXPLAIN command. It tells you exactly how the database plans to run your query. You should look for "Full Table Scans" (bad for large tables) and ensure the database is using the "Index Seek" (good) operations you expect.


EXPLAIN SELECT name, salary FROM employees WHERE dept_id = 10;
-- The output shows if an index was used, 
-- how many rows were scanned, and the join type.

3. Advanced Indexing Strategies

  • Composite Indexes: An index on multiple columns. Order matters! An index on (last_name, first_name) is useful for searching by last name, but NOT for searching by first name alone.
  • Covering Indexes: An index that contains all the columns requested in the SELECT clause. This allows the DBMS to answer the query using only the index, without ever touching the actual table (very fast).
  • Functional Indexes: An index on the result of a function (e.g., UPPER(email)).

Real-World Relationship: The GPS and the Library Search

Think of the Query Optimizer like a GPS Navigation App. You give it a destination (your SQL Query). The GPS looks at multiple routes: the highway (Index), the side streets (Full Table Scan), or the toll road (In-Memory Cache). It calculates the "Cost" in time and fuel and picks the fastest one. Sometimes, if the highway is under construction (Missing Statistics), the GPS might accidentally send you down a slow dirt road. Think of an Index like the Dewey Decimal System in a library. Without it, you'd have to walk past every single book (Full Table Scan) to find "Harry Potter." With the index, you know exactly which floor, which aisle, and which shelf to go to, saving you miles of walking (Disk I/O).
