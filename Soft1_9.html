Software Engineering I: Page 9 - Design Patterns: Behavioral Patterns

The Architecture of Logic

In our previous pages, we learned how objects are created and how they are structured. Behavioral Patterns focus on how objects communicate and distribute responsibilities. They are concerned with the assignment of responsibilities between objects and the algorithms used to accomplish complex tasks. By using behavioral patterns, you can make the workflow of your software more flexible, decoupled, and easy to change without having to rewrite the entire system logic.

1. Observer Pattern

The Observer pattern defines a one-to-many dependency between objects. When one object (the Subject) changes state, all its dependents (the Observers) are notified and updated automatically. This is the foundation of event-driven programming and modern UI frameworks like React or Vue.

The Code implementation (Java)


interface Observer { void update(String news); }

class NewsAgency {
    private List observers = new ArrayList<>();
    public void addObserver(Observer o) { observers.add(o); }
    public void setNews(String news) {
        for (Observer o : observers) o.update(news); // Notify everyone
    }
}

2. Strategy Pattern

The Strategy pattern defines a family of algorithms, encapsulates each one, and makes them interchangeable. This allows the algorithm to vary independently from the clients that use it. Instead of a giant if-else block for different behaviors, you swap out the "Strategy" object at runtime.

The Code implementation (Java)


interface PaymentStrategy { void pay(int amount); }

class CreditCardPay implements PaymentStrategy { public void pay(int a) { /* ... */ } }
class CryptoPay implements PaymentStrategy { public void pay(int a) { /* ... */ } }

class ShoppingCart {
    private PaymentStrategy strategy;
    public void setStrategy(PaymentStrategy s) { this.strategy = s; }
    public void checkout(int a) { strategy.pay(a); }
}

3. Command Pattern

The Command pattern encapsulates a request as an object, thereby letting you parameterize clients with different requests, queue or log requests, and support Undoable operations. It decouples the object that invokes the operation from the one that knows how to perform it.


interface Command { void execute(); void undo(); }

class LightOnCommand implements Command {
    private Light light;
    public void execute() { light.turnOn(); }
}

Real-World Relationship: The Newspaper, the Payment App, and the Remote Control

Think of the Observer like a Newspaper Subscription. The newspaper company (the Subject) doesn't care exactly what you're doing. Whenever they have a new edition (a State Change), they just send it to every person on the subscriber list (the Observers). If you want to stop getting news, you just "detach" from the list. Think of the Strategy like a Modern Payment App. When you go to pay, you choose a strategy: Apple Pay, Credit Card, or Bitcoin. The app (the Client) doesn't change its core logic; it just uses whichever "Plugin" (the Strategy) you selected for that specific transaction. Finally, think of the Command like a Universal Remote Control. Each button is a "Command Object." The remote doesn't know what "Channel Up" does; it just knows that when you press that button, it should tell the "TV" (the Receiver) to execute the command. This allows the remote to work with a TV, a Stereo, or a DVD player without needing separate logic for each.
