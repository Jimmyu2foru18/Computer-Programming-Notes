Software Engineering I: Page 6 - The Engineering Mindset: Clean Code (DRY, KISS, YAGNI)

Beyond Syntax: Writing for Humans

In the professional world, code is rarely written once and forgotten. It is read, maintained, and extended by dozens of people over many years. As Martin Fowler famously said: "Any fool can write code that a computer can understand. Good programmers write code that humans can understand." This page explores the core design heuristics that separate professional software engineers from hobbyist programmers. These principles help us fight "Technical Debt" and "Complexity Bloat."

1. DRY: Don't Repeat Yourself

Every piece of knowledge or logic in a system should have a single, unambiguous representation. When you copy-paste code, you create a maintenance nightmare. If you find a bug in one copy, you have to remember to fix it in all five copies. DRY encourages Modularity and Abstaction.

The Code implementation (Java)


// BAD: Logic repeated in multiple places
void validateUser(String email) {
    if (email == null || !email.contains("@")) throw new Error();
}
void validateAdmin(String email) {
    if (email == null || !email.contains("@")) throw new Error();
}

// GOOD: Single point of truth
class Validator {
    static void checkEmail(String email) {
        if (email == null || !email.contains("@")) throw new Error();
    }
}

2. KISS: Keep It Simple, Stupid

Complexity is the enemy of reliability. Engineers should always strive for the simplest solution that solves the problem. Don't use a complex Design Pattern if a simple if statement will do. Simple code is easier to test, harder to break, and faster to explain to a teammate.

3. YAGNI: You Aren't Gonna Need It

Never add functionality until it is actually required. Developers often try to "future-proof" code by adding features they think they might need later. This leads to wasted time and a bloated codebase. Only build what you need now. If you need more later, the OCP (Open/Closed Principle) will help you extend it.

4. SoC: Separation of Concerns

A system should be organized so that each part addresses a separate concern. Your UI code should not know how the database works; your Database code should not know how to send emails. This results in Low Coupling and High Cohesion.

Real-World Relationship: The IKEA Manual and the Professional Kitchen

Think of Clean Code like a Good IKEA Manual. It uses simple diagrams (KISS) and clear steps. It doesn't tell you how to build a chair if you're currently building a desk (YAGNI). If a screw is used in 10 different places, they don't give it 10 different names; it's just "Part #101" (DRY). Think of SoC like a Professional Restaurant Kitchen. The pastry chef handles dessert, the saucier handles sauces, and the dishwasher handles cleanup. They don't switch roles in the middle of a rush. Each person (Module) has a specific "Concern," ensuring the kitchen functions like a well-oiled machine rather than a chaotic brawl.
