Managing Background Services

In Linux, background processes that run without a user interface are called Services (or daemons). From your web server (Apache/Nginx) to your database (MySQL/PostgreSQL), almost every essential system function runs as a service. Modern Linux distributions use systemd as the standard service manager. It is responsible for starting services during boot, monitoring their health, and restarting them if they crash. Understanding systemctl and how to create custom service files is the key to running your own applications as robust, production-grade Linux services.

1. Managing Services with systemctl

The systemctl command is your primary tool for service management:

  • systemctl status <service>: Check if it's running.
  • systemctl start/stop/restart <service>: Basic control.
  • systemctl enable/disable <service>: Configure whether it starts automatically at boot.

# Ensure Nginx starts automatically
sudo systemctl enable nginx
# Start it now
sudo systemctl start nginx

2. Creating Custom Services

To run your own application as a service, create a unit file in /etc/systemd/system/. This file tells systemd how to run your program, which user to run it as, and what to do if it fails.


[Unit]
Description=My Custom App
[Service]
ExecStart=/usr/bin/python3 /opt/myapp/app.py
Restart=always
[Install]
WantedBy=multi-user.target

Real-World Relationship: The Hotel Night Manager

Think of systemd like the Night Manager of a Hotel. They are responsible for making sure everything is running smoothly while the guests (the users) are asleep. They ensure the security staff (firewall service) is on duty, the HVAC (web server service) is working, and the emergency lights (database service) are on. If a lightbulb burns out (a service crashes), the Night Manager notices and replaces it immediately (auto-restart). Creating a custom service file is like giving the manager a detailed set of instructions on how to take care of a new feature you've just installed in the hotel.
