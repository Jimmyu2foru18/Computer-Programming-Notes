Software Engineering I: Page 7 - Design Patterns: Creational Patterns

Solving Common Problems

In software engineering, a Design Pattern is a general, reusable solution to a commonly occurring problem within a given context. Patterns are not finished designs that can be transformed directly into code; they are templates for how to solve a problem that can be used in many different situations. Creational Patterns focus on object creation mechanisms, trying to create objects in a manner suitable to the situation. They increase flexibility and reuse of existing code.

1. Singleton Pattern

The Singleton ensures that a class has only one instance and provides a global point of access to it. This is useful for things like database connection pools or configuration managers where multiple copies would be wasteful or dangerous.

The Code implementation (Java)


public class Database {
    private static Database instance;
    private Database() {} // Private constructor prevents 'new'

    public static Database getInstance() {
        if (instance == null) {
            instance = new Database();
        }
        return instance;
    }
}

2. Factory Method Pattern

The Factory Method defines an interface for creating an object, but lets subclasses decide which class to instantiate. This follows the DIP (Dependency Inversion Principle) by letting you work with abstractions rather than concrete classes.

The Code implementation (Java)


interface Shape { void draw(); }
class Circle implements Shape { public void draw() { /* circle */ } }
class Square implements Shape { public void draw() { /* square */ } }

class ShapeFactory {
    public Shape getShape(String type) {
        if (type.equalsIgnoreCase("CIRCLE")) return new Circle();
        if (type.equalsIgnoreCase("SQUARE")) return new Square();
        return null;
    }
}

3. Builder Pattern

The Builder pattern separates the construction of a complex object from its representation. It allows you to produce different types and representations of an object using the same construction code. This is perfect for objects with many optional parameters (the "Telescoping Constructor" anti-pattern).


Computer myPC = new Computer.Builder()
    .setCPU("Intel i7")
    .setRAM("16GB")
    .setGPU("RTX 3080")
    .build();

Real-World Relationship: The High-End Restaurant and the Subway Sandwich

Think of the Singleton like the Executive Head Chef of a world-famous restaurant. There is only one Head Chef. If you need an answer about the menu, everyone in the building goes to that one person. If there were two Head Chefs giving different orders, the kitchen would fall into chaos. Think of the Factory Method like a Restaurant Menu. You order "Pasta." You don't care exactly which cook makes it or how they boil the water; the "Factory" (the Kitchen) just hands you a "Pasta" object that satisfies your hunger. Finally, think of the Builder like a Subway Sandwich Shop. You don't just order "Food." You walk down the line: "I want white bread," then "add turkey," then "no onions," then "toasted." You build your unique sandwich step-by-step until it's exactly what you need.
