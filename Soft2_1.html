Software Engineering II: Page 1 - System Modeling: UML Structural Diagrams

The Blueprint of Software

In architecture, you wouldn't start building a skyscraper without a set of blue-printed plans. In software engineering, we use the Unified Modeling Language (UML) to visualize, specify, and document our systems. UML is not a programming language; it is a visual language that allows engineers to communicate complex architectures across different teams and technologies. Structural Diagrams focus on the static aspects of a system—the classes, objects, and components that exist, and how they are physically or logically related to one another.

1. The Class Diagram: The Master Blueprint

The Class Diagram is the most used diagram in UML. It describes the structure of a system by showing the system's classes, their attributes, their methods, and the relationships between them. A class is represented as a rectangle with three sections: name, attributes (data), and operations (methods).

Key Relationships:

  • Association: A simple link between two classes (e.g., a Teacher "teaches" a Student).
  • Aggregation (Hollow Diamond): A "has-a" relationship where the parts can exist independently (e.g., a Car has a Spare Tire).
  • Composition (Filled Diamond): A strong "has-a" relationship where the parts cannot exist without the whole (e.g., a House has a Wall).
  • Generalization (Arrow): An "is-a" relationship (Inheritance).
  • Realization (Dashed Arrow): Interface implementation.

2. Component Diagrams: High-Level View

While Class Diagrams focus on the "code level," Component Diagrams focus on the "system level." They show how the system is divided into modular physical components (like a database, an API, or a third-party library) and the dependencies between them. This is essential for understanding the overall architecture and how different services interact.

3. Object Diagrams: A Snapshot in Time

An Object Diagram is essentially a "snapshot" of a Class Diagram at a specific moment during runtime. It shows actual instances of classes and their current values. This is helpful for debugging complex logic where the relationships between specific data instances are causing issues.

Real-World Relationship: The Architect's Drawing and the Model Home

Think of a Class Diagram like an Architect's Drawing for a new suburb. It defines where the houses go, what every kitchen must look like, and how the streets connect. It is a set of rules. Think of an Object Diagram like a Model Home in that suburb. It is one real, physical instance of the blueprint. You can walk inside, see the furniture (Attributes), and see how the living room connects to the kitchen. Finally, think of Composition like the Rooms in the House: if the house is demolished, the rooms cease to exist. Aggregation is like the Furniture: you can sell the couch (Independent part) and the house still stands, or you can move the couch to a different house.
