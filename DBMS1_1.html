DBMS I: Page 1 - Introduction to Database Management Systems

The Evolution of Data Storage

In the early days of computing, data was stored in simple flat files. Each application had its own set of files, leading to massive redundancy, inconsistency, and difficulty in data retrieval. A Database Management System (DBMS) is the sophisticated software layer that solved these problems. It provides a systematic way to create, retrieve, update, and manage data. By acting as an intermediary between the user and the physical data, a DBMS ensures that information is stored efficiently, remains consistent across multiple users, and is protected from unauthorized access.

Why Not Just Use a File System?

  • Data Redundancy: File systems often store the same information in multiple places, wasting space.
  • Data Inconsistency: If a customer's address changes in one file but not another, the data becomes unreliable.
  • Concurrency Issues: If two people try to edit the same file at once, data can be lost.
  • Security: File systems lack the granular access control (e.g., allowing a user to see names but not salaries) that a DBMS provides.

The 3-Schema Architecture

One of the most important concepts in database design is Data Independence. This is achieved through the Three-Schema Architecture (or ANSI-SPARC architecture), which separates the user's view from the physical storage.

  1. External Level (User View): What the user sees. Different users can have different "views" of the same data (e.g., an HR view vs. a Finance view).
  2. Conceptual Level (Logical View): The "Blueprint" of the entire database. It defines what data is stored and the relationships between tables, without worrying about how it's physically saved.
  3. Internal Level (Physical View): How the data is actually stored on the hard drive (indexes, hashing, block sizes).

DBMS Languages: DDL, DML, and DCL

To interact with a DBMS, we use specialized languages. Most modern databases use SQL (Structured Query Language), which is divided into several sub-languages:

  • DDL (Data Definition Language): Used to define the structure (Schema). Commands: CREATE, ALTER, DROP.
  • DML (Data Manipulation Language): Used to work with the data itself. Commands: SELECT, INSERT, UPDATE, DELETE.
  • DCL (Data Control Language): Used to manage permissions and security. Commands: GRANT, REVOKE.

Real-World Relationship: The Library and the Spreadsheet

Think of a File System like a Pile of Unorganized Folders on a desk. If you want to find a specific client, you have to look through every folder. If you have two folders for the same client, you might update one and forget the other (Inconsistency). Think of a DBMS like a State-of-the-Art Public Library. You don't just walk in and look at books; you use a "Catalog" (the Schema). You can search by Title, Author, or Genre (Querying). The "Librarian" (the DBMS) ensures that if one person checks out a book, no one else can take it at the same time (Concurrency). The External Schema is like the Books on the Shelves—it's what you see and interact with. The Internal Schema is the Library's Storage Warehouse in the basement—you don't need to know how the boxes are stacked to find the book you want.
