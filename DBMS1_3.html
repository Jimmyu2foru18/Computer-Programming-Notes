DBMS I: Page 3 - SQL Basics: Data Definition & Database Keys

Structured Query Language (SQL)

In our previous pages, we explored the theory of the relational model. Now, we move into the practical world of SQL (Structured Query Language). SQL is the standard language used to communicate with relational databases like MySQL, PostgreSQL, and Oracle. While different databases have their own small variations (dialects), the core SQL syntax is universal. This page focuses on Data Definition Language (DDL)—the commands used to create the physical structure of your database—and the critical concept of Keys, which define the integrity and relationships of your data.

1. Defining Structure: DDL Commands

DDL is used to build the "Empty Boxes" that will hold your data. When you run a DDL command, you are modifying the Data Dictionary (the database's internal metadata).

Core Commands:

  • CREATE TABLE: Defines a new table, its columns, and their data types.
  • ALTER TABLE: Modifies an existing table (e.g., adding a new column or changing a data type).
  • DROP TABLE: Deletes the table and all its data permanently.

CREATE TABLE employees (
    emp_id INT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    salary DECIMAL(10, 2),
    hire_date DATE
);

2. Database Keys: The Anchors of Integrity

In a relational database, you must have a way to uniquely identify every row and connect tables together. This is achieved through Keys.

  • Primary Key (PK): A unique identifier for every row. It cannot be NULL and no two rows can share the same PK. Usually an ID number.
  • Foreign Key (FK): A column that "points" to the Primary Key of another table. This is how we create Relationships between data.
  • Composite Key: A primary key that consists of two or more columns combined (e.g., in a "Course Enrollment" table, the StudentID and CourseID together might be the PK).
  • Unique Key: Ensures that all values in a column are unique, but unlike a PK, it allows a single NULL value.

3. Speeding Up Data: Indexes

An Index is a special data structure (usually a B-Tree) that the DBMS uses to find rows faster. Without an index, the database has to scan every single row in a table to find one record. With an index, it can jump straight to the correct location. However, indexes make INSERT and UPDATE operations slower, as the index must be rebuilt every time data changes.


CREATE INDEX idx_employee_name ON employees(name);

Real-World Relationship: The ID Badge and the Index of a Book

Think of a Primary Key like your Social Security Number or your Employee ID Badge. There might be 1,000 people named "John Smith," but each has a unique ID number that distinguishes them. Think of a Foreign Key like a Reference on a Resume. The resume doesn't contain all the information about your previous boss; it just contains their "Manager ID" (the FK), which "points" back to the main Employee table where their full details are stored. Finally, think of an Index like the Index at the back of a Textbook. If you want to find "Normalization," you don't read the whole book from page 1. You go to the index, find "N," see the page number, and flip straight there. The index makes your search much faster, but it takes up extra pages in the book (Storage) and must be updated if the page numbers change (Write overhead).
