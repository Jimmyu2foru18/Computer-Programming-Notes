Java I: Page 8 - Modular Logic: Methods & Overloading

Grouping Action

In the first seven pages, we learned how to store data and control the flow of execution. However, if we simply kept writing code in one long main method, our programs would become unreadable. To build professional software, we must break our logic into smaller, reusable blocks called Methods (known as "functions" in other languages). Methods allow you to write a specific set of instructions once and "call" them from anywhere else in your program, following the DRY (Don't Repeat Yourself) principle.

1. Method Syntax

In Java, every method must be part of a class. The syntax is strict and requires you to define the visibility, the return type, and the parameters.


// [access] [modifier] [return_type] [name]([parameters])
public static int addNumbers(int a, int b) {
    int sum = a + b;
    return sum; // Hand the result back to the caller
}
  • Access Modifier: (e.g., public) Determines who can call this method.
  • Return Type: (e.g., int) The type of data the method sends back. Use void if the method returns nothing.
  • Parameters: The inputs the method needs to do its job.

2. Method Overloading

In many languages, you can't have two functions with the same name. Java, however, allows Method Overloading. You can have multiple methods with the exact same name, as long as they have different Signatures (different types or number of parameters). The compiler is smart enough to know which one you want based on the arguments you pass.


public int multiply(int a, int b) { return a * b; }

public double multiply(double a, double b) { return a * b; }

public int multiply(int a, int b, int c) { return a * b * c; }

3. Varargs: Variable Arguments

Sometimes you don't know how many inputs a method will receive. Java handles this with Varargs (type... name), which treats the incoming arguments as an array inside the method.


public int sumAll(int... numbers) {
    int total = 0;
    for (int n : numbers) total += n;
    return total;
}

Real-World Relationship: The ATM and the Recipe

Think of a Method like a Recipe in a cookbook. The recipe is written once. It has placeholders like "Quantity of sugar" (Parameters). When you decide to bake (Call the method), you provide the actual sugar (Arguments). The cake you pull out of the oven is the Return Value. Think of Method Overloading like a Smart ATM. If you put in a card, it shows the "Withdraw" menu. If you put in a check, it shows the "Deposit" menu. The ATM (the Method Name) is the same, but it changes its behavior based on what you "give" it (the Arguments). This flexibility makes the machine intuitive and easy to use without needing separate buttons for every tiny variation.
