Java II: Page 9 - Concurrency: Multithreading Basics

Doing Many Things at Once

In most of our previous examples, code ran sequentially—one line after another. But in the real world, software often needs to handle multiple tasks at the same time. A web server must handle hundreds of users simultaneously; a video game must update graphics while playing sound and checking for input. Multithreading is the ability of a CPU to provide multiple threads of execution concurrently. Java was built with multithreading as a core feature, making it a primary choice for high-performance, concurrent applications.

1. Threads: The Unit of Execution

A Thread is the smallest unit of processing that can be performed in an OS. In Java, you can create and manage threads using the Thread class. There are two primary ways to define a thread's task.

Method A: Extending the Thread Class


class MyThread extends Thread {
    public void run() {
        System.out.println("Thread is running!");
    }
}
// Usage
MyThread t1 = new MyThread();
t1.start(); // Spawns the new thread

Method B: Implementing Runnable (Preferred)

This is generally preferred because it allows your class to still inherit from another class (since Java only supports single inheritance).


class MyTask implements Runnable {
    public void run() {
        System.out.println("Task is running in a thread!");
    }
}
// Usage
Thread t2 = new Thread(new MyTask());
t2.start();

2. Thread Lifecycle and Control

A thread goes through several states: New, Runnable, Running, Blocked/Waiting, and Terminated. You can control a thread using methods like sleep() (pause execution) and join() (wait for a thread to finish).


try {
    Thread.sleep(1000); // Pause for 1 second
} catch (InterruptedException e) {
    e.printStackTrace();
}

3. Synchronization: Preventing Chaos

When multiple threads try to access the same data (like a bank balance) at the same time, data corruption can occur. This is called a Race Condition. Java uses the synchronized keyword to ensure that only one thread can access a specific piece of code at a time.

Real-World Relationship: The Restaurant Kitchen and the Shared Notebook

Think of Multithreading like a Busy Restaurant Kitchen. If there was only one chef (one Thread), the customers would wait hours for their food. Instead, there are multiple chefs working concurrently. One chops vegetables, one grills steak, and one plates the food. They all work independently but contribute to the same meal (the Program). Think of a Race Condition like two chefs trying to write in the same inventory notebook at the same time—the ink gets smudged and the count becomes wrong. Synchronization is like a Lock on the Notebook: only one chef can hold the pen at a time. They must wait for the other to finish before they can record their data, ensuring the information remains clean and accurate.
