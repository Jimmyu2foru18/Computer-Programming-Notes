Web Technologies II: Page 6 - Security: Web Best Practices

The Security Mindset

In web development, you should operate under the assumption that every user is a potential attacker. Whether malicious or just misguided, user input is the most common vector for hacks. Web security is not a "phase" you complete before launch; it is a mindset you apply to every single line of code you write. The top three vulnerabilities—XSS, SQL Injection, and CSRF—can be devastating, but they are all preventable with disciplined coding habits.

1. The "Big Three" Vulnerabilities

  • XSS (Cross-Site Scripting): An attacker injects malicious JavaScript into your page, which then runs in other users' browsers. They can steal cookies, redirect users, or deface your site. Defense: Never trust user input. Sanitize all data before displaying it (e.g., use htmlspecialchars() in PHP or DOM text methods in JS).
  • SQL Injection: An attacker inserts malicious SQL code into form fields to manipulate your database queries. Defense: ALWAYS use prepared statements (PDO in PHP, Mongoose queries in Node).
  • CSRF (Cross-Site Request Forgery): An attacker tricks an authenticated user into performing an action they didn't intend (e.g., changing their password or transferring funds). Defense: Use unique, cryptographically secure anti-CSRF tokens for every form submission.

// Secure PHP example against XSS
echo htmlspecialchars($userInput, ENT_QUOTES, 'UTF-8');

2. Defense in Depth

Security isn't about one "silver bullet." You need layered defenses: HTTPS (encryption), Content Security Policy (CSP) headers, rate limiting (to stop brute force), and secure authentication/authorization (hashing passwords with bcrypt, never storing them in plain text).

Real-World Relationship: The Secure Bank Teller

Think of Web Security like a Fortress. Your application is the interior, and your users are the people entering. Sanitization is the security guard checking everyone for hidden weapons (malicious scripts) before they enter. Prepared Statements are the secure procedure that ensures a visitor can only fill out a "Withdrawal Slip" to take *their own* money, not to command the teller to empty the whole vault. CSRF Tokens are like a Secret Handshake that only the trusted teller and the authorized customer know. If an attacker tries to trick the teller, they won't know the secret handshake, and the teller will refuse to act.
