Software Engineering I: Page 2 - OOP Pillars: Encapsulation & Inheritance

The Foundation of Modern Design

Object-Oriented Programming (OOP) is the dominant paradigm in software engineering. It is a way of designing software that models real-world entities as "Objects." These objects bundle together State (attributes) and Behavior (methods). By organizing code this way, we can build massive systems that are modular, reusable, and easy to understand. To master OOP, you must understand its four pillars. This page covers the first two: Encapsulation and Inheritance.

1. Encapsulation: The Protector of Data

Encapsulation is the practice of bundling data and the methods that operate on that data into a single unit (a Class) and restricting direct access to the internal state. In engineering terms, this is called "Information Hiding." By hiding the internal details, you prevent outside code from accidentally breaking your object's data. You provide a controlled interface (public methods) while keeping the implementation (private variables) hidden.

The Code implementation (Java)


public class BankAccount {
    private double balance; // Private: Hidden from the world

    public double getBalance() {
        return balance;
    }

    public void deposit(double amount) {
        if (amount > 0) {
            balance += amount; // Internal logic is protected
        }
    }
}

Benefits: Data integrity, flexibility (you can change internal logic without breaking external code), and security.

2. Inheritance: The Legacy of Code

Inheritance is a mechanism where a new class (the Subclass) derives properties and behaviors from an existing class (the Superclass). This promotes Code Reuse and establishes a logical hierarchy. It represents an "Is-A" relationship. For example, a Manager is an Employee. A Truck is a Vehicle.

The Code implementation (Java)


class Animal {
    String name;
    void eat() { System.out.println("Animal is eating..."); }
}

class Dog extends Animal { // Dog inherits 'name' and 'eat()'
    void bark() { System.out.println("Woof!"); }
}

Real-World Relationship: The ATM and the Smartphone

Think of Encapsulation like an ATM. You interact with the machine through a keypad and a screen (the Interface). You don't have access to the cash vault, the mechanical gears, or the bank's internal ledger (the Implementation). The machine protects its "State" (the money) from you, ensuring you only interact with it in valid ways. Think of Inheritance like Smartphone Evolution. An iPhone 15 inherits the core traits of the original iPhone (a touchscreen, the ability to call, an operating system). Apple didn't reinvent the concept of a "mobile phone" from scratch; they extended the existing blueprint, adding new sensors and faster processors while keeping the "Legacy" features that worked.
