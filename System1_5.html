System Design I: Page 5 - State Management

Managing Complexity

In a large application, data is shared between dozens of components. Keeping that data in sync—ensuring a change in the user profile updates the navbar, the profile page, and the settings page simultaneously—is the central challenge of State Management. As applications grow, passing data down through nested components (prop drilling) becomes impossible. State management libraries provide a centralized store, allowing any component to access or update global state efficiently.

1. The State Management Pattern

Modern state management (like Redux or Zustand) often follows a Unidirectional Data Flow: Actions trigger state changes, which update the store, and the store updates the UI.


// Example: Simple store with Zustand
import create from 'zustand';
const useStore = create((set) => ({
  count: 0,
  inc: () => set((state) => ({ count: state.count + 1 })),
}));

2. Why Centralized State?

  • Consistency: The "source of truth" is in one place.
  • Predictability: State changes are usually traced and logged (e.g., Redux DevTools).
  • Performance: Only components that explicitly "subscribe" to a piece of state re-render when it changes.

Real-World Relationship: The Central Warehouse

Think of Component-level state like a Small Kitchen Pantry in each apartment of a building. It's fast, but you have to go to every single apartment to check if they have milk. Think of Centralized State Management like a Central Warehouse for the whole building. Every apartment (component) just checks the central warehouse (the Store) to see how much milk is available. If an apartment buys more milk, they update the warehouse records, and every other apartment immediately sees the updated inventory count.
