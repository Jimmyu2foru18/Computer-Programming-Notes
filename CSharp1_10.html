C# I: Page 10 - Introduction to OOP: Classes & Objects

The Blueprint for Reality

In the first nine pages of this course, we focused on "Procedural" programming—writing instructions and methods that work on data. However, as software becomes more complex, it is often more natural to organize our code around Objects. This style of programming is called Object-Oriented Programming (OOP). In OOP, we group related data (State) and behaviors (Methods) together into a single unit called a Class. This allows us to model real-world things—like a User, a Bank Account, or a Game Character—directly in our code. This page concludes the Foundations module by exploring how to define your own types and bring them to life as objects.

1. The Class vs. The Object

To create objects, we first need a Class. A class is a blueprint or a template. It defines what data an object will have and what it can do, but it doesn't contain the data itself. An Object (or Instance) is the actual thing built from that blueprint. You can use one class to create thousands of unique objects.

CLASS: Dog (Blueprint)
Fields: Name, Breed
Methods: Bark()
→
OBJECT: myDog (Instance)
Name: "Buddy"
Breed: "Golden Retriever"

2. Defining a Class in C#

A class typically contains Fields (variables) and Methods. We use the new keyword to "Instantiate" the class and create an object in memory.


class Dog {
    // State (Data)
    public string Name;
    public string Breed;

    // Behavior (Action)
    public void Bark() {
        Console.WriteLine($"{Name} says: Woof!");
    }
}

// Creating an instance
Dog myDog = new Dog();
myDog.Name = "Buddy";
myDog.Bark(); // Output: Buddy says: Woof!

3. Constructors: The Birth of an Object

A Constructor is a special method that runs automatically the moment you create an object. Its job is to set the initial state of the object. Constructors have the exact same name as the class and no return type.


class Dog {
    public string Name;

    // Constructor
    public Dog(string name) {
        this.Name = name;
    }
}
// Now we must provide a name when we create the dog:
Dog d = new Dog("Rex");

Real-World Relationship: The Cookie Cutter and the Passport

Think of a Class like a Cookie Cutter. The cutter itself isn't a cookie; it's just a metal shape that defines what a cookie looks like. Think of the Object as the actual Cookie you bake. You can use the same cutter to make 100 cookies, but each cookie is a separate "Instance." You can put chocolate chips on one and sprinkles on another—they all have the same shape (the Class), but they have different details (the State). Another example is a Passport Office: the blank application form is the Class. Once the office prints the actual Passport with your photo and name on it, that unique book is the Object. It has the same structure as every other passport, but it belongs uniquely to you.
