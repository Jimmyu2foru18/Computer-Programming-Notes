Software Engineering I: Page 5 - SOLID Design Principles: LSP, ISP & DIP

Completing the SOLID Foundation

In the previous page, we established that a class should have one job (SRP) and should be easy to extend without modification (OCP). This page covers the final three principles of SOLID: LSP, ISP, and DIP. These principles deal with the relationships between classes and how to structure dependencies to avoid "Fragile" and "Rigid" code. When followed correctly, they allow large teams to work on a single codebase with confidence that their changes won't break unrelated features.

3. Liskov Substitution Principle (LSP)

"Objects of a superclass should be replaceable with objects of its subclasses without affecting the correctness of the program."

This means a subclass must truly "Is-A" version of its parent. It shouldn't break the parent's logic or throw unexpected exceptions for methods the parent is supposed to handle. A classic violation is a Penguin class inheriting from a Bird class that has a fly() method.

The Code implementation (Java)


// BAD: Penguin can't fly!
class Bird { void fly() { /* ... */ } }
class Penguin extends Bird {
    void fly() { throw new UnsupportedOperationException(); }
}

// GOOD: Proper hierarchy
class Bird { /* general bird traits */ }
class FlyingBird extends Bird { void fly() { /* ... */ } }
class Sparrow extends FlyingBird { /* ... */ }
class Penguin extends Bird { void swim() { /* ... */ } }

4. Interface Segregation Principle (ISP)

"No client should be forced to depend on methods it does not use."

Instead of one giant "Fat" interface, it is better to have many small, specialized interfaces. This prevents a class from having to implement "dummy" methods that it doesn't actually need.

The Code implementation (Java)


// BAD: Fat Interface
interface SmartDevice { void print(); void fax(); void scan(); }

// GOOD: Segregated Interfaces
interface Printer { void print(); }
interface Scanner { void scan(); }

class BasicPrinter implements Printer {
    public void print() { /* implementation */ }
}

5. Dependency Inversion Principle (DIP)

"Depend on abstractions, not concretions."

High-level modules (like your business logic) should not depend on low-level modules (like a specific database). Both should depend on an Interface. This allows you to swap out the low-level module (e.g., switching from MySQL to MongoDB) without ever touching your high-level logic.

The Code implementation (Java)


// GOOD: Depending on Abstraction
interface MessageService { void send(String msg); }

class EmailService implements MessageService { 
    public void send(String msg) { /* send email */ } 
}

class UserNotification {
    private MessageService service; // Depends on interface, NOT EmailService
    public UserNotification(MessageService s) { this.service = s; }
    void notifyUser() { service.send("Hello!"); }
}

Real-World Relationship: The Battery and the Wall Socket

Think of LSP like Batteries. If you have a remote control that takes "AA" batteries, you can use a Duracell, an Energizer, or a rechargeable Amazon brand. As long as they follow the "AA Specification" (the Superclass), the remote doesn't care which one you use—it will work correctly. Think of ISP like a Power Strip. You don't want a power strip that forces you to plug in a toaster, a lamp, and a fridge all at once just to get power for your phone. You want separate outlets (Interfaces) so you only plug in what you need. Finally, think of DIP like a Wall Socket. Your house provides a standardized "Abstraction" (the Socket). Your lamp (High-level) doesn't know if the power is coming from a Coal Plant, a Solar Farm, or a Wind Turbine (Low-level Details). If the power company changes their "Implementation" (switches from Coal to Solar), your lamp keeps working without any changes because it depends on the Abstraction of the wall socket.
