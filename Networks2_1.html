Networks II: Page 1 - TCP: Connection-Oriented Reliability

The Trustworthy Protocol

In our previous module, we learned how data is routed across networks using IP. However, IP is "best-effort"—it doesn't guarantee that a packet will arrive, nor does it guarantee they will arrive in the right order. This is why we need the Transmission Control Protocol (TCP). TCP sits on top of IP at the Transport Layer (Layer 4) and provides a reliable, ordered, and error-checked byte stream. If you are browsing the web, checking email, or transferring files, you are using TCP. It turns the chaotic nature of the internet into a dependable communication channel.

1. The Three-Way Handshake

Before any data can be exchanged, TCP must establish a connection. This is done through a "three-way handshake," which ensures that both the sender and the receiver are ready to communicate.

The Handshake Steps:

  1. SYN (Synchronize): The client sends a packet with the SYN flag set to the server, suggesting a connection.
  2. SYN-ACK (Synchronize-Acknowledge): The server responds with a SYN-ACK packet, confirming it received the request and is ready to talk.
  3. ACK (Acknowledge): The client sends an ACK packet, confirming it received the server's response. The connection is now established.

Client              Server
  |-- SYN ----------->|
  |<-- SYN-ACK -------|
  |-- ACK ----------->|
  | Connection Established |

2. Ensuring Reliability: Sequence & Acknowledgments

Once connected, TCP breaks data into Segments. Each segment gets a Sequence Number. If the receiver gets segment #1, #3, and #4, it knows segment #2 is missing and asks the sender to retransmit it. This ensures data arrives in the correct order, even if packets take different paths across the internet.

Flow & Congestion Control

  • Flow Control: Prevents the sender from overwhelming the receiver's buffer (using a "Window Size").
  • Congestion Control: Detects if the network itself is overloaded (packet loss) and slows down the transmission rate to prevent a network-wide collapse.

Real-World Relationship: The Registered Letter

Think of TCP like a Registered Letter (Certified Mail). You write a letter, and the post office gives you a tracking number. They guarantee it will arrive. When the recipient gets the letter, they must sign for it (Acknowledgment). If the letter doesn't arrive in three days, you get a notification that the delivery failed and you send a second copy. Think of UDP, on the other hand, like Sending a Bunch of Postcards. You drop 100 postcards in the mailbox. Some might get there before others, some might be delivered in reverse order, and some might fall out of the mail truck and disappear. You don't get tracking numbers, but the postcards were much faster and cheaper to send. TCP is for when the *content* matters; UDP is for when *speed* matters.
