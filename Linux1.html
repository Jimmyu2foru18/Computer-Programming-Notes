Linux Guide: Deep Dive into Kernel and System Internals

1. The Linux Kernel and Architecture

The Linux kernel is a monolithic kernel, meaning core services, drivers, and the network stack reside within the kernel address space, enabling high performance through direct hardware access. To manage different hardware, it utilizes a modular approach where drivers can be loaded/unloaded as modules.

The Virtual Filesystem (VFS) Switch

A crucial architectural feature is the VFS. It provides an abstraction layer that allows the kernel to interact with various physical filesystems (ext4, XFS, Btrfs, NFS) through a unified set of operations (open, read, write). User-space applications do not need to know the underlying filesystem type; the VFS handles the translation.

Kernel vs. User Space

User-space applications cannot access hardware directly. When a program needs to perform I/O, allocate memory, or spawn a process, it issues a system call (syscall). This triggers a hardware interrupt, switching the processor from User Mode to Kernel Mode, executing the privileged kernel function, and then switching back.

2. Process Scheduling: CFS and Interrupt Handling

At the heart of process management is the Completely Fair Scheduler (CFS). CFS utilizes a Red-Black tree to maintain a balance of "virtual runtime" (vruntime) across all tasks. Processes with lower vruntime are prioritized to ensure fairness over long-term execution.

Interrupts and SoftIRQs

Hardware devices issue hard interrupts, forcing the CPU to immediately stop the current task to handle device input (e.g., a packet arriving on a NIC). To keep systems responsive, the kernel immediately performs only the most critical work in the Top Half (Interrupt Handler) and schedules the remainder of the work in the Bottom Half using SoftIRQs or Tasklets, which are processed later in kernel context.

3. Advanced Filesystem Mechanics and Inodes

Filesystems are the structures used to store and organize data on block devices.

Inode Structures

An inode (index node) is a data structure on the filesystem that stores all metadata about a file (permissions, owner, file size, timestamps, and pointers to the physical data blocks), but not the filename or the file data itself. Filenames are stored in directory files, which map names to inode numbers.

Journaling (ext4) and CoW (Btrfs)

Journaling (ext4) ensures integrity by logging changes before committing them to the filesystem metadata. Conversely, Btrfs uses Copy-on-Write (CoW). Instead of updating data in place, it writes the new data to a new block, then atomically updates the metadata pointer. This enables efficient, read-only snapshots and instant backups.
