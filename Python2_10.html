Writing Code That Doesn't Break

In the professional world, "working code" is not enough. Your code must be robust—able to handle unexpected input without crashing—and it must be testable—verifiable by automated systems to ensure that future changes don't break existing features. This final page covers the two pillars of software quality: Exception Handling and Unit Testing.

Exception Handling: The Safety Net

An Exception is an error that occurs while a program is running. If you don't "catch" the exception, your program will terminate immediately. Using the try...except block, you can intercept these errors and decide how to respond.

The Full try Structure

  • try: The code you want to run.
  • except: Code that runs if an error occurs.
  • else: Code that runs ONLY if no error occurred.
  • finally: Code that runs no matter what (perfect for cleanup).

try:
    number = int(input("Enter a number to divide 100 by: "))
    result = 100 / number
except ValueError:
    print("That wasn't a valid number!")
except ZeroDivisionError:
    print("You can't divide by zero!")
else:
    print(f"Success! The result is {result}")
finally:
    print("Division attempt complete.")

Automated Quality: Unit Testing

A Unit Test is a piece of code that tests a specific "unit" of your program (usually a function or a method). Python's built-in unittest library allows you to create test suites that you can run with a single command to prove your code works as expected.

Example: Testing a Calculator


import unittest

def add(a, b): return a + b

class TestCalculator(unittest.TestCase):
    def test_add_positive(self):
        self.assertEqual(add(2, 3), 5)
    
    def test_add_negative(self):
        self.assertEqual(add(-2, -3), -5)

if __name__ == '__main__':
    unittest.main()

Real-World Relationship: The Safety Net and the Inspector

Think of Exception Handling like a Trapeze Safety Net. The circus performers (your code) are doing dangerous, high-stakes work. Most of the time they land perfectly. But if they slip (an error), the net catches them so the show can go on, rather than letting them hit the ground (Crashing the program). Think of Unit Testing like a Quality Control Inspector in a car factory. Every single bolt and engine part (each Function) is tested individually before the car is ever put together. If a bolt fails the test, the inspector sends it back to be fixed. This ensures that by the time a customer drives the car, thousands of mini-tests have already proven that it is safe and reliable.
