Python II: Page 5 - Object-Oriented Programming (OOP): Encapsulation & Properties

The Principle of Data Protection

In the previous page, we learned how to build objects with data and behavior. However, simply having data isn't enough—we must also protect it. Encapsulation is the practice of bundling data and methods into a single unit and restricting direct access to the internal state of the object. This prevents accidental corruption of data and allows the developer to control how information is modified.

Access Control: Public, Protected, and Private

Unlike languages like Java, Python does not have strict "private" keywords. Instead, it relies on community naming conventions to signal how an attribute should be treated.

1. Public (Default)

Attributes like self.name are public. They can be read and modified from anywhere.

2. Protected (_attribute)

A single underscore is a "soft" signal to other developers: "This is internal; please don't touch it directly." Python won't stop you, but you are breaking the contract of the class.

3. Private (__attribute)

A double underscore triggers Name Mangling. Python changes the name of the attribute internally (e.g., _ClassName__attribute) to make it much harder to access from outside the class.


class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner          # Public
        self._balance = balance     # Protected
        self.__pin = "1234"         # Private (Mangled)

The Pythonic Way: @property Decorators

If we make an attribute private, we still need a way to interact with it. In other languages, you write get_balance() and set_balance() methods. In Python, we use the @property decorator to make methods behave exactly like attributes. This provides a clean interface while allowing for validation behind the scenes.


class BankAccount:
    def __init__(self, initial_balance=0):
        self._balance = initial_balance

    @property
    def balance(self):
        """The Getter."""
        return self._balance

    @balance.setter
    def balance(self, amount):
        """The Setter with validation."""
        if amount >= 0:
            self._balance = amount
        else:
            raise ValueError("Balance cannot be negative!")

Real-World Relationship: The ATM and the Volume Knob

Think of Encapsulation like an ATM. The bank has a huge "Database" of money (Attributes), but they don't let you reach into the vault and grab cash yourself (Direct Access). Instead, they provide a screen and keypad (The Interface). You can only withdraw() or deposit() by following their rules and providing a PIN. Think of the @property Setter like the Volume Knob on a high-end stereo. To you, it's just a simple dial. But inside, the knob is a "Setter" that has validation: it won't let you turn the volume to 1,000% because that would blow the speakers. The complexity of the electrical circuit is hidden (Encapsulated), leaving you with a safe and simple way to control the system.
