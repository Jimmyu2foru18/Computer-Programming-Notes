System Design I: Page 4 - Frontend: Components

Modular UI Architecture

In modern web development (React, Vue, Angular), the old practice of creating massive, monolithic HTML pages is obsolete. Instead, we use a Component-Based Architecture. A component is an independent, reusable UI piece that encapsulates its own structure (HTML), style (CSS), and logic (JavaScript). This approach transforms a complex web application into a collection of simple, manageable "building blocks."

1. Why Components?

  • Reusability: Build a button or navbar once, use it everywhere.
  • Maintainability: Change the code in one component, and it updates everywhere that component is used.
  • Testability: Smaller components are much easier to unit test than large pages.

// Example: A reusable button component (React)
const CustomButton = ({ label, onClick }) => (
  <button onClick={onClick} className="btn">
    {label}
  </button>
);

2. Component Anatomy

A component typically consists of three parts:

  1. Props: Data passed *into* the component from its parent (read-only configuration).
  2. State: Internal data managed *by* the component (e.g., whether a toggle is on or off).
  3. Lifecycle: Hooks that allow you to run code when the component is created, updated, or destroyed.

Real-World Relationship: The Lego Building Block

Think of a Monolithic page like a one-piece plastic toy—if you want to change one part of it, you have to replace the whole thing. Think of a Component-Based Architecture like a Lego Set. You build the house out of individual, standardized blocks. If you want to change the color of the roof, you don't rebuild the whole house; you just swap out the red roof bricks for blue ones. This is the power of component-based systems: standardized, reusable, and modular.
