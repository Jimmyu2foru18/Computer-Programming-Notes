DBMS II: Page 2 - Advanced Concepts: Concurrency Control & Locking

Managing Multi-User Chaos

In a modern enterprise, a database is rarely used by one person at a time. Thousands of web requests, mobile apps, and internal tools are hitting the database simultaneously. If two users try to update the same record at the exact same time, the data can become corrupted. Concurrency Control is the set of techniques used by a DBMS to manage simultaneous access to data without compromising Isolation (the 'I' in ACID). The most common method for achieving this is through Locking, which ensures that only one transaction can modify a specific data item at a time.

1. The Mechanisms of Locking

A lock is a variable associated with a data item that describes the status of that item with respect to possible operations that can be applied to it.

  • Shared Lock (S): Also called a "Read Lock." Multiple transactions can hold a shared lock on the same item at the same time. You can read the data, but you cannot change it.
  • Exclusive Lock (X): Also called a "Write Lock." Only one transaction can hold an exclusive lock. No other transaction can read or write to that item until the lock is released.

-- Manually requesting an exclusive lock (standard in many SQL dialects)
SELECT * FROM products WHERE p_id = 1 FOR UPDATE;

2. Transaction Isolation Levels

Not every application needs perfect isolation. Sometimes, we sacrifice a little consistency for a massive boost in speed. SQL standard defines four isolation levels:

  1. Read Uncommitted: No locks. You can see data that hasn't been committed yet (Dirty Reads). Fastest but most dangerous.
  2. Read Committed: You only see data once it is committed. (The default for most databases).
  3. Repeatable Read: Ensures that if you read a row twice in one transaction, the data won't change in between.
  4. Serializable: Perfect isolation. Transactions run as if they were in a single-file line. Slowest but most consistent.

3. The Deadlock Problem

A Deadlock occurs when Transaction A is waiting for a lock held by Transaction B, while Transaction B is waiting for a lock held by Transaction A. They are both stuck forever. Modern DBMSs have a Deadlock Detector that periodically checks for these "cycles" and automatically kills one of the transactions to let the other proceed.

Real-World Relationship: The Shared Office and the Four-Way Stop

Think of Shared and Exclusive Locks like a Shared Document in an Office. If the document is "Read-Only" (Shared Lock), everyone can look at it at the same time. But if one person wants to "Edit" it (Exclusive Lock), they must "Check it out," and no one else can even look at it until they check it back in. Think of a Deadlock like a Four-Way Stop Sign where four cars arrive at the exact same time. Every driver is waiting for the person to their right to go first. If no one takes initiative, they will sit there forever. Finally, think of Isolation Levels like a Security Guard at a construction site. Read Uncommitted is no guard—anyone can walk in and see the half-finished walls. Serializable is a guard who only lets one person enter the site at a time, ensuring they see a perfectly finished room, but creating a huge line at the gate.
