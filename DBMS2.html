Database Management Systems: Advanced Topics in Distributed and Analytical Systems

1. Distributed Databases, Consensus, and PACELC

Distributed databases achieve scale by spreading data across nodes. The CAP Theorem is the baseline, but PACELC is more comprehensive: if there is a Partition, how does the system trade-off between Availability and Consistency; Else (when running normally), how does it trade-off between Latency and Consistency?

Consensus Algorithms: Raft Deep Dive

In CP (Consistent and Partition-tolerant) systems, consensus algorithms like Raft enable nodes to agree on state. Raft decomposes consensus into leader election, log replication, and safety, ensuring only the most up-to-date node can become the new leader.

2. NoSQL: Storage Engine Internals and Bloom Filters

LSM-Trees are optimized for write-heavy workloads. To optimize read performance, DBMSs use Bloom Filters—a probabilistic structure that tells you if an element is *definitely not* in a set, preventing unnecessary disk I/O.

3. Scaling Strategies

As applications grow, single-node databases become bottlenecks.

  • Vertical Scaling: Increasing CPU/RAM on one machine.
  • Horizontal Scaling (Sharding): Distributing data across multiple machines using a shard key.
  • Replication: Using read-replicas to offload query traffic.
  • Connection Pooling: Reusing connections to reduce latency.

4. Top Concepts for Backend Interviews

  • B-Tree vs. Hash Indexes: Equality vs. Range Scans.
  • N+1 Query Problem: Performance pitfalls in ORMs.
  • Deadlocks: Prevention via consistent locking order.
  • Primary/Foreign Keys: Referential integrity.
  • Query Execution Plans: Debugging with EXPLAIN.
  • Caching: Using Redis/Memcached.

5. The Expert Mindset: Query Optimization

Expert developers treat performance as an iterative process:

  1. Identify: Use EXPLAIN ANALYZE to find slow components.
  2. Evaluate: Check if the query performs sequential scans (bad) or index scans (good).
  3. Refine: Add indexes, optimize joins, and avoid SELECT *.
  4. Monitor: Proactively track slow query logs.

6. OLTP vs. OLAP: The Storage Divide

The architecture changes based on transactional vs. analytical workloads. OLAP (Columnar) storage is dramatically faster for analytical aggregates as only required columns are read from disk.
