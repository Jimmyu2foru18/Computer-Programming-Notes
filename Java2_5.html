Java II: Page 5 - Object-Oriented Programming (OOP): Abstract Classes

The Middle Ground of Abstraction

In our previous pages, we looked at concrete classes (which have everything implemented) and interfaces (which traditionally have nothing implemented). But what if you have a group of classes that share some logic, but also have unique behaviors that must be implemented? For this, Java provides the Abstract Class. An abstract class is a restricted class that cannot be used to create objects. It serves as a base for other classes, providing a mix of Concrete Methods (with code) and Abstract Methods (without code).

1. Defining an Abstract Class

You use the abstract keyword for both the class and any methods that don't have an implementation. If a class contains at least one abstract method, the class itself must be declared abstract.


public abstract class Vehicle {
    protected String brand;

    public Vehicle(String brand) {
        this.brand = brand;
    }

    // Concrete Method (Shared logic)
    public void displayBrand() {
        System.out.println("Brand: " + brand);
    }

    // Abstract Method (Must be implemented by children)
    public abstract void startEngine();
}

2. Implementing Abstract Classes

To use an abstract class, you must inherit from it using extends. The subclass is then responsible for providing the "body" for all abstract methods inherited from the parent.


public class Car extends Vehicle {
    public Car(String brand) {
        super(brand);
    }

    @Override
    public void startEngine() {
        System.out.println("Turning the key and starting the combustion engine...");
    }
}

3. Abstract Classes vs. Interfaces

This is one of the most common questions in Java development. Which one should you use?

  • Use an Interface if you want to define a Contract that can be implemented by many unrelated classes (e.g., Comparable, Serializable).
  • Use an Abstract Class if you want to share Code and State (fields) among closely related classes (e.g., all Vehicles have a brand and an engine).

Real-World Relationship: The Half-Finished Car and the Template

Think of an Abstract Class like a Car Chassis on an assembly line. It has the frame, the wheels, and the basic electronics (Concrete methods/Logic). You can't "drive" a chassis on the street (you can't Instantiate it). However, it serves as the base for building a "Sedan," a "Truck," or an "SUV" (the Subclasses). Each of those models adds its own unique body and interior (Abstract methods) but they all share the same underlying frame. Another example is a Government Form Template: the template defines where the name and date go, but it is "Abstract" until you fill it out with your specific information (Implementation). The template itself isn't a valid legal document—it's just the blueprint for one.
