Software Engineering II: Page 7 - System Design: Architecture & Microservices

The Evolution of Structure

System Architecture is the high-level structure of a software system—it is the "map" that shows how different components are organized and how they communicate. In the early days of software, most applications were Monoliths, meaning the entire system was built as a single, massive unit. While simple to deploy, monoliths become difficult to scale and maintain as they grow. Today, the industry has shifted toward Microservices—an architectural style that structures an application as a collection of small, autonomous services modeled around a business domain.

1. Monolith vs. Microservices

Choosing an architecture is a series of trade-offs. There is no "perfect" structure; there is only the structure that best fits your team's size and project goals.

  • Monolith: Fast to develop initially, easy to test, and simple to deploy. However, a single bug can crash the whole system, and you must scale the entire app even if only one feature is slow.
  • Microservices: Each service is independent. You can use different languages for different tasks (e.g., Python for AI, Java for Finance). If one service fails, the others keep running. However, it is much more complex to manage and requires strong DevOps skills.

2. Service Communication

In a microservices world, services must talk to each other to complete a task. There are two primary patterns:

  1. Synchronous (REST/gRPC): Service A calls Service B and waits for an immediate answer. This is simple but creates tight coupling.
  2. Asynchronous (Message Queues): Service A sends a message to a "Queue" (like RabbitMQ or Kafka) and moves on. Service B picks up the message whenever it's ready. This is the foundation of Event-Driven Architecture.

3. Key Components: The API Gateway

In a complex system with 50 microservices, you don't want the user's browser to talk to all 50 directly. An API Gateway acts as a single entry point. it handles security, logging, and "routes" the user's request to the correct internal service. It is the "Concierge" of your system architecture.

Real-World Relationship: The General Store and the Mega-Mall

Think of a Monolith like a Small-Town General Store. One building holds the food, the tools, the clothes, and the post office. It's easy to manage—you only have one set of keys and one roof to fix. But if the roof leaks over the tools, the whole store might have to close. Think of Microservices like a Mega-Mall. The "Food Court" is one service, the "Clothing Store" is another, and the "Cinema" is a third. They are all in different buildings or sections. If the Food Court has a grease fire (a Service Failure), the Cinema can stay open. You can hire a different manager for each store (Technology Agnostic). However, you now need a security team, a central information desk (API Gateway), and a complex walk-way system to get people between the stores (Networking).
