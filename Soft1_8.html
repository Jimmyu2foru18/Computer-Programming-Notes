Software Engineering I: Page 8 - Design Patterns: Structural Patterns

The Architecture of Connection

While Creational Patterns handle how objects are born, Structural Patterns deal with how objects and classes are composed to form larger, more complex structures. These patterns help ensure that when one part of a system changes, the entire structure doesn't collapse. They are the "connectors" and "wrappers" that allow incompatible systems to work together and allow for the dynamic expansion of functionality without massive rewrites.

1. Adapter Pattern

The Adapter allows classes with incompatible interfaces to work together. It acts as a bridge between two objects, converting the interface of one class into an interface expected by the client. This is essential for integrating third-party libraries or legacy code into a modern system.

The Code implementation (Java)


// Target interface
interface MediaPlayer { void play(String fileName); }

// Adaptee (Legacy/Incompatible)
class VlcPlayer { void playVlc(String fileName) { /* logic */ } }

// Adapter
class MediaAdapter implements MediaPlayer {
    VlcPlayer vlcPlayer = new VlcPlayer();
    public void play(String fileName) { vlcPlayer.playVlc(fileName); }
}

2. Decorator Pattern

The Decorator pattern allows you to attach new behaviors to objects by placing these objects inside special wrapper objects that contain the behaviors. This is a flexible alternative to subclassing (Inheritance) for extending functionality, following the OCP (Open/Closed Principle).

The Code implementation (Java)


interface Coffee { double getCost(); }
class SimpleCoffee implements Coffee { public double getCost() { return 2.0; } }

// Decorator
class MilkDecorator implements Coffee {
    private Coffee coffee;
    public MilkDecorator(Coffee c) { this.coffee = c; }
    public double getCost() { return coffee.getCost() + 0.5; }
}

3. Facade Pattern

The Facade pattern provides a simplified interface to a complex set of classes, a framework, or a library. Instead of a client having to interact with dozens of small, complex objects, they interact with one "Facade" object that handles the complexity behind the scenes.


class HomeTheaterFacade {
    private Amp amp;
    private Projector projector;

    public void watchMovie() {
        amp.on();
        projector.on();
        projector.wideScreenMode();
    }
}

Real-World Relationship: The Power Adapter and the Car Dashboard

Think of the Adapter like a Physical Travel Plug. Your American hair dryer (the Client) can't plug into a European socket (the Service) directly because the pins don't match. You use an adapter that has American holes on one side and European pins on the other. Think of the Decorator like Ordering a Pizza. You start with a base (the Simple Object). Then you add pepperoni, then mushrooms, then extra cheese. Each topping is a "Decorator" that wraps the base pizza and adds "value" (Cost and Taste) without needing a separate class for "PizzaWithMushroomsAndCheese." Finally, think of the Facade like the Ignition Key in a car. To start a car, hundreds of things happen: fuel is injected, sparks fly, the battery engages, and the pistons move. You don't do those things individually. You just turn the key (the Facade), and the car handles the complex subsystem for you.
