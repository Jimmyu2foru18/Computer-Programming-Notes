Software Engineering II: Page 9 - Enterprise Standards: Security & Code Quality

The Fortress Mindset

In modern software engineering, security is not a "feature" you add at the end; it must be built into the foundation of your code. Every line of code you write is a potential doorway for an attacker. Defensive Programming is the practice of writing code under the assumption that "things will go wrong" and "people will try to break it." Quality software is not just bug-free—it is secure, resilient, and adheres to strict professional standards that protect both the user's data and the company's integrity.

1. Common Security Vulnerabilities

To defend a system, you must first understand how it is attacked. The OWASP Top 10 is the standard awareness document for developers about the most critical web application security risks.

SQL Injection (SQLi)

This occurs when an attacker inserts malicious SQL code into an input field, allowing them to manipulate your database. Prevention: Never use string concatenation for queries. Always use Prepared Statements or an ORM.


// BAD: Vulnerable to SQLi
String query = "SELECT * FROM users WHERE email = '" + userInput + "'";

// GOOD: Secure parameterized query
PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE email = ?");
ps.setString(1, userInput);

Cross-Site Scripting (XSS)

Attackers inject malicious scripts into web pages viewed by other users. Prevention: Always sanitize and escape user-provided data before displaying it on a page.

2. Principles of Secure Design

  • Least Privilege: Every module, process, or user should only have the minimum permissions necessary to do its job. A "Web Scraper" doesn't need "Delete" permissions on your main database.
  • Defense in Depth: Layer your security. Don't just rely on a firewall; use encryption, input validation, and secure authentication at every level.
  • Fail Securely: If an error occurs (e.g., an exception), the system should default to its most secure state (e.g., denying access) rather than leaving a "backdoor" open.

Real-World Relationship: The Bank Vault and the ID Check

Think of Input Validation like a Security Guard at a Bank. If you try to hand the guard a grenade instead of a withdrawal slip, they stop you immediately. You don't let the "Grenade" (the Malicious Input) into the building. Think of Parameterized Queries like a Pre-Formatted Form. You can only write your name in the designated "Name" box. You can't write a command like "Give me everyone's money" in that box, because the system is only looking for characters that make up a name. Finally, think of Least Privilege like a Janitor's Keyring. The janitor has keys to the closets and the offices, but they don't have the keys to the main cash vault. They have exactly what they need to do their job, and nothing more, which limits the damage if their keys are stolen.
