DBMS II: Page 6 - Data Scaling: Distributed Databases & Sharding

Scaling Beyond One Server

A single database server, no matter how powerful, has a physical limit. It can only handle so many queries per second and store so many terabytes of data. For global platforms like Amazon or Google, a single server is impossible. They use Distributed Databases, where data is spread across multiple physical locations. This provides Fault Tolerance (if one server burns down, the app stays up) and Extreme Scalability. This page explores the three primary techniques for spreading data: Fragmentation, Replication, and the modern standard of Sharding.

1. Fragmentation: Slicing the Tables

Fragmentation is the process of breaking a single table into smaller "Fragments" and storing them on different servers.

  • Horizontal Fragmentation (Sharding): Slicing the table by Rows. (e.g., Users 1-1,000,000 on Server A, Users 1,000,001+ on Server B).
  • Vertical Fragmentation: Slicing the table by Columns. (e.g., Public profiles on Server A, private billing info on a high-security Server B).

2. Replication: Copying the Data

Replication is the process of storing the exact same data on multiple servers. This is mainly used for Availability and Read Performance.

  • Master-Slave Replication: All "Writes" (Updates) go to the Master server. The Master then "pushes" the changes to multiple Slaves. Read requests are spread across all the Slaves.
  • Master-Master Replication: Every server can handle both reads and writes. This is more complex because you must handle "Conflicts" if two people update the same record on different servers simultaneously.

3. Sharding: The Modern Standard

Sharding is a specific type of horizontal partitioning. You choose a Shard Key (like user_id or country). The database uses a mathematical formula (Hashing) on that key to decide which server a piece of data belongs to. This allows you to add 1,000 servers and spread the load perfectly across them.

Real-World Relationship: The Encyclopedia and the Chain Restaurant

Think of Sharding like a Set of Encyclopedias. You have 26 books (the Shards). If you want to find "Zebra," you don't look through 25 other books; you go straight to the "Z" book. This allows 26 people to look up words at the exact same time without fighting over one book. Think of Replication like a Chain Restaurant (McDonald's). They have the exact same menu and ingredients (Data) at 30,000 locations. If the McDonald's in your town is closed for repairs (Server Failure), you can just go to the next town over and get the same burger. The "System" is always available because the data is replicated everywhere.
