DBMS II: Page 1 - Advanced Concepts: Transactions & ACID Properties

The Unit of Reliable Work

In our foundations module, we learned how to store and query data. Now, we enter the advanced world of Transaction Management. In a busy database system with thousands of users, simple SQL commands aren't enough. Imagine you are transferring money between two bank accounts. You must subtract from one and add to the other. If the system crashes after the subtraction but before the addition, the money vanishes. A Transaction is a logical unit of work that contains one or more SQL statements. It must be treated as a single, atomic operation to ensure the database remains in a valid state. This reliability is guaranteed by the four ACID Properties.

1. The ACID Properties: The Pillar of Trust

  • Atomicity: "All or Nothing." If any part of the transaction fails, the entire transaction is rolled back, and the database remains unchanged.
  • Consistency: A transaction must move the database from one consistent state to another, following all rules and constraints.
  • Isolation: Concurrent transactions must not interfere with each other. Each transaction should feel like it is the only one running on the system.
  • Durability: Once a transaction is "Committed," its changes are permanent and will survive even a power outage or system crash.

2. Transaction Control: COMMIT and ROLLBACK

In SQL, you manage transactions using specific control commands. A transaction begins with the first DML statement and ends with either a COMMIT (save changes) or a ROLLBACK (undo changes).


START TRANSACTION;

UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;

-- If everything is okay:
COMMIT;

-- If an error occurred:
ROLLBACK;

3. The Write-Ahead Log (WAL)

How does a database ensure Durability? It uses a Write-Ahead Log. Before any data is actually changed in the main database files, the DBMS writes the intended change to a sequential log on the disk. If the system crashes, the DBMS reads this log upon restart and "replays" the committed transactions that hadn't made it to the main files yet.

Real-World Relationship: The Bank Transfer and the Unfinished Email

Think of Atomicity like Sending an Email with an Attachment. If the internet cuts out while the attachment is 50% uploaded, the email isn't sent with half a photo—the whole thing fails, and you have to start over. It's all or nothing. Think of Isolation like Two People Booking the Same Airplane Seat. Even if they click "Buy" at the exact same millisecond, the DBMS ensures that the first person's transaction finishes (Seating them) before the second person's transaction even looks at the seat's availability. This prevents the "Race Condition" where two people think they own the same seat. This level of control is what makes digital commerce possible and trustworthy.
