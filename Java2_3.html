Java II: Page 3 - Object-Oriented Programming (OOP): Polymorphism & Overriding

The Art of Many Forms

In the previous pages, we learned how objects are built and how they inherit traits. Now, we come to the crown jewel of Object-Oriented Programming: Polymorphism. The word comes from Greek, meaning "many forms." In Java, it refers to the ability of an object to take on multiple identities depending on the context. This allows you to write code that is incredibly flexible, scalable, and easy to extend without changing existing logic.

1. Runtime Polymorphism: Method Overriding

Runtime polymorphism (or Dynamic Method Dispatch) is achieved through Method Overriding. This occurs when a subclass provides a specific implementation for a method that is already defined in its superclass. The decision of which method to call is made by the JVM at runtime, based on the actual type of the object, not the type of the reference.


class Animal {
    public void makeSound() { System.out.println("Generic sound"); }
}

class Dog extends Animal {
    @Override // Good practice: catches typos!
    public void makeSound() { System.out.println("Woof!"); }
}

class Cat extends Animal {
    @Override
    public void makeSound() { System.out.println("Meow!"); }
}

2. Upcasting: Reference vs. Object

The magic of polymorphism happens when we use a Superclass Reference to point to a Subclass Object. This is called Upcasting.


Animal myPet = new Dog(); // Upcasting
myPet.makeSound();        // Output: Woof! (The JVM knows it's a Dog)

Animal anotherPet = new Cat();
anotherPet.makeSound();   // Output: Meow!

3. Compile-time Polymorphism: Method Overloading

As we saw in Module 1, Method Overloading allows multiple methods in the same class to have the same name but different signatures. This is "Static Polymorphism" because the compiler knows which method to call at compile-time based on the arguments provided.

Real-World Relationship: The Universal Remote and the Electrical Outlet

Think of Polymorphism like a Universal TV Remote. The remote has a "Volume Up" button. It doesn't need to know if your TV is a Sony, a Samsung, or an LG—it just sends the "Volume Up" command (the Method Call), and each TV responds in its own specific way (the Overridden Implementation). Think of Method Overloading like a Swiss Army Knife: it's one tool, but it has different "forms" (a blade, a screwdriver, a saw). You "call" the knife based on which form you pull out (the Arguments). Finally, think of Method Overriding like Custom Car Tuning: the factory gives you a base engine (the Superclass), but you override the exhaust and the turbo (the Subclass) to give it a unique sound and performance, while still being able to drive it like any other car on the road.
