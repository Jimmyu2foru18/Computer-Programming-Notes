DBMS I: Page 10 - Data Integrity: Constraints, Triggers & Views

Guarding the Data

In our final page of DBMS Foundations, we explore the mechanisms that ensure data remains Accurate, Consistent, and Secure. Simply having a normalized structure isn't enough; we need automated rules that prevent invalid data from entering the system and abstractions that hide complexity from the end user. Constraints define the "Rules of Engagement" for your tables, Triggers provide automated reactions to data changes, and Views offer a tailored, simplified perspective of the database. Together, these tools form the "Inner Circle" of database security and integrity.

1. Entity and Referential Integrity

We've already seen Primary and Foreign Keys, which enforce the most basic integrity rules. But SQL allows for even more granular control:

  • NOT NULL: Ensures a column must have a value (e.g., every user must have an email).
  • UNIQUE: Prevents duplicate values in a column.
  • CHECK Constraints: Enforces a specific condition on the data. For example, CHECK (price > 0) ensures you never have a negative price.
  • DEFAULT: Provides a fallback value if none is provided.

CREATE TABLE products (
    p_id INT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    price DECIMAL(10,2) CHECK (price > 0),
    stock INT DEFAULT 0
);

2. Automation with Triggers

A Trigger is a stored procedure that automatically runs in response to a specific event on a table (like an INSERT, UPDATE, or DELETE). Triggers are useful for maintaining audit logs, calculating derived values, or enforcing complex business rules that simple constraints can't handle.


CREATE TRIGGER audit_salary_change
AFTER UPDATE ON employees
FOR EACH ROW
BEGIN
    INSERT INTO salary_history(emp_id, old_sal, new_sal)
    VALUES (OLD.id, OLD.salary, NEW.salary);
END;

3. Data Abstraction: Views

A View is a virtual table based on the result-set of an SQL query. It doesn't store data itself; it just provides a window into one or more underlying tables. Views are essential for Security (you can show a user the names but hide the salaries) and Simplicity (you can hide a complex 5-table join behind a single view name).


CREATE VIEW employee_summary AS
SELECT e.name, d.dept_name, e.job_title
FROM employees e
JOIN departments d ON e.dept_id = d.id;

-- Now users can just query the view!
SELECT * FROM employee_summary;

Real-World Relationship: The Security Guard and the Secret Compartment

Think of Constraints like the Rules at an Airport Security Gate. "No liquids over 3oz" (a CHECK constraint). "Everyone must have a ticket" (NOT NULL). If you try to break a rule, you aren't allowed through the gate (the Data is rejected). Think of a Trigger like a Security Camera that automatically sends an alert to the manager if it sees a door being forced open. It reacts instantly to the event. Finally, think of a View like a Secret Compartment in a Desk. You (the Admin) know exactly how the whole desk is built, but you only show your guest (the User) the one drawer they are allowed to use. They don't need to see the complex structural support or the private documents hidden in the other drawers.
