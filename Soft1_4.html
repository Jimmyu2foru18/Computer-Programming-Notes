Software Engineering I: Page 4 - SOLID Design Principles: SRP & OCP

The Blueprint for Maintainable Code

Object-Oriented Programming gives us the tools, but SOLID Principles give us the rules. Introduced by Robert C. Martin (Uncle Bob), SOLID is a set of five design principles intended to make software designs more understandable, flexible, and maintainable. In the professional world, code is read and modified much more often than it is written. Following SOLID ensures that a change in one part of the system doesn't cause a "house of cards" collapse in another. This page covers the first two letters: SRP and OCP.

1. Single Responsibility Principle (SRP)

"A class should have only one reason to change."

This means a class should do one thing, and do it well. If a class is responsible for both calculating a salary AND saving that salary to a database, it has two responsibilities. If you change your database provider, you have to modify the salary-calculation class—this is a design flaw.

The Code implementation (Java)


// BAD: Multiple responsibilities
class Employee {
    void calculatePay() { /* logic */ }
    void saveToDatabase() { /* database logic */ }
}

// GOOD: Separated concerns
class Employee {
    void calculatePay() { /* logic */ }
}
class EmployeeRepository {
    void save(Employee e) { /* database logic */ }
}

Benefits: Easier testing, fewer bugs, and higher "Cohesion."

2. Open/Closed Principle (OCP)

"Software entities should be open for extension, but closed for modification."

You should be able to add new functionality to a system without changing the existing code. This is usually achieved through Abstraction and Polymorphism. Instead of using a giant switch statement to handle new types, you use an interface.

The Code implementation (Java)


// BAD: Modifying existing code for every new type
class PaymentProcessor {
    void process(String type) {
        if (type.equals("Credit")) { /* ... */ }
        else if (type.equals("PayPal")) { /* ... */ }
    }
}

// GOOD: Extend by adding new classes
interface PaymentMethod { void pay(); }
class CreditCard implements PaymentMethod { public void pay() { /* ... */ } }
class PayPal implements PaymentMethod { public void pay() { /* ... */ } }

class PaymentProcessor {
    void process(PaymentMethod method) { method.pay(); }
}

Real-World Relationship: The Swiss Army Knife and the Wall Outlet

Think of SRP like a Swiss Army Knife. Each tool has one job: the scissors cut, the screwdriver turns screws, the blade slices. You don't have one giant "Do-Everything" tool because if the screwdriver tip breaks, you'd have to throw away the whole knife. By having separate tools (Classes), you can fix or sharpen one without affecting the others. Think of OCP like a Standardized Wall Outlet. The outlet is "Closed for Modification"—you don't have to tear apart the wall and rewire the house every time you buy a new appliance. However, it is "Open for Extension"—you can plug in a lamp, a computer, or a vacuum, as long as they follow the standard plug "Interface." You add new power-consuming functionality (Extension) without ever touching the house's electrical system (Modification).
