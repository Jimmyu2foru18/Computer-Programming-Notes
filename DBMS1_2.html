DBMS I: Page 2 - The Relational Model & Relational Algebra

The Dominant Data Model

In the previous page, we saw the general purpose of a DBMS. While there are many ways to organize data (Hierarchical, Network, Object-Oriented), the Relational Model is by far the most widely used in the world today. Introduced by E.F. Codd in 1970, it is based on the mathematical concept of a Relation. In this model, data is organized into two-dimensional tables consisting of rows and columns. This simplicity is what makes it so powerful, as it allows for intuitive data management and mathematically sound querying.

1. Relational Terminology

To speak like a database engineer, you must master the formal terminology of the relational model:

  • Relation (Table): A collection of related data.
  • Tuple (Row): A single record in the relation (e.g., one specific employee).
  • Attribute (Column): A named property of the relation (e.g., "Salary" or "Email").
  • Domain: The set of allowable values for an attribute (e.g., "Age" must be an integer between 0 and 120).
  • Degree: The total number of attributes (columns) in a relation.
  • Cardinality: The total number of tuples (rows) in a relation.

2. Relational Algebra: The Logic of Data

Relational Algebra is a theoretical language that provides a set of operations for manipulating relations. It is the "Math" that powers the SQL language we use every day. Every SQL query you write is internally converted into a series of relational algebra operations by the database engine.

Basic Operations:

  • Selection (σ): Filters rows based on a condition (e.g., find all employees with salary > 50000).
  • Projection (π): Selects specific columns and discards the rest (e.g., show only "Names" and "Emails").
  • Union (∪): Combines all rows from two tables (they must have the same schema).
  • Intersection (∩): Returns only the rows present in both tables.
  • Difference (-): Returns rows in the first table that are not in the second.
  • Cartesian Product (×): Combines every row from the first table with every row from the second.

Real-World Relationship: The Ledger and the Filter

Think of a Relation like a Physical Business Ledger. The book is the table. Each page (Tuple) represents one transaction. Each column (Attribute) represents a detail like "Date," "Amount," or "Client." Relational Algebra is like the Office Clerk working with that ledger. Selection is the clerk saying, "Give me only the pages from January." Projection is the clerk saying, "Copy only the client names and ignore the amounts." Join (a complex operation) is like the clerk looking at the Customer Ledger and the Sales Ledger and matching the Customer ID to find out which city bought the most products. This logical, mathematical approach ensures that data retrieval is predictable and accurate every single time.
